The Complete Guide to Angular Testing and a Weird Trick for Faster Unit Tests (2019) Panduan Lengkap untuk Pengujian Sudut dan Trik Aneh untuk Tes Unit Lebih Cepat (2019)

July 15, 2018 15 Juli 2018 By Christian 7 Comments Oleh Christian 7 Komentar

When doing practices like continuous delivery and continuous integration, testing becomes a more important part of the work because it's unreasonable to regression test all possible effects of a release manually. Ketika melakukan praktik seperti pengiriman terus menerus dan integrasi berkelanjutan, pengujian menjadi bagian yang lebih penting dari pekerjaan karena tidak masuk akal untuk menguji semua efek yang mungkin dari rilis secara manual. When doing five releases on a workday, you don't get to do a full manual regression testing on every release as that would easily take up one person's time for a day. Ketika melakukan lima rilis pada hari kerja, Anda tidak bisa melakukan pengujian regresi manual penuh pada setiap rilis karena itu akan dengan mudah menghabiskan waktu satu orang selama sehari. Instead, you want your code to be nicely covered with the right tests for ensuring the application work as expected. Alih-alih, Anda ingin kode Anda ditutupi dengan baik dengan tes yang tepat untuk memastikan aplikasi berfungsi seperti yang diharapkan.

What are the different kinds of tests? Apa sajakah jenis tes?

There are different ways to test in your application and you should always ask yourself: “What kind of test will cover this feature” and then choose that test. Ada berbagai cara untuk menguji dalam aplikasi Anda dan Anda harus selalu bertanya pada diri sendiri: "Tes seperti apa yang akan mencakup fitur ini" dan kemudian pilih tes itu.

The testing pyramid Piramida pengujian

You might have heard about the testing pyramid and the saying that 80 % of the tests should be unit tests, 15 % is integration tests and 5 % is UI tests. Anda mungkin pernah mendengar tentang piramida pengujian dan mengatakan bahwa 80% dari tes harus unit test, 15% adalah tes integrasi dan 5% adalah tes UI. These percentages might vary depending on who you ask but the ratio should be around the same. Persentase ini mungkin bervariasi tergantung pada siapa yang Anda tanyakan tetapi rasionya harus sama. Note also, that we get static typing in Angular apps, which can be seen as part of the unit tests coverage, as I don't recommend we unit test every component with 100% code coverage, but instead focus on the essential areas: services, pure functions, and container components. Perhatikan juga, bahwa kita mendapatkan pengetikan statis di aplikasi Angular, yang dapat dilihat sebagai bagian dari cakupan pengujian unit, karena saya tidak menyarankan kita menguji setiap komponen dengan cakupan kode 100%, tetapi lebih fokus pada area penting: layanan , fungsi murni, dan komponen wadah.

When going up the testing pyramid, tests become more expensive because; Ketika naik piramida pengujian, tes menjadi lebih mahal karena; they are harder to create, they are harder to maintain/more flakey because they test on fragile interfaces like the UI. mereka lebih sulit untuk dibuat, mereka lebih sulit untuk mempertahankan / lebih serpihan karena mereka menguji pada antarmuka yang rapuh seperti UI. On the top of the pyramid is manual testing, which still has its place, even in the modern ages of automatization and DevOps, because validating a new feature for the first time requires a human to decide if it is passing the user acceptance criteria (UAT) . Di atas piramida adalah pengujian manual, yang masih memiliki tempatnya, bahkan di zaman modern otomatisasi dan DevOps, karena memvalidasi fitur baru untuk pertama kalinya membutuhkan manusia untuk memutuskan apakah ia melewati kriteria penerimaan pengguna (UAT) ) . In Angular context, there are the following types of test: isolated and shallow unit testing , integration tests between components and UI/E2E tests , which can be functional and visual regression testing. Dalam konteks sudut, ada jenis tes berikut: pengujian unit terisolasi dan dangkal , tes integrasi antara komponen dan tes UI / E2E , yang dapat berupa pengujian regresi fungsional dan visual.

Isolated unit testing Pengujian unit terisolasi

Isolated tests are used to test a unit in isolation. Tes terisolasi digunakan untuk menguji unit secara terpisah. The unit might contain some business logic that needs to be tested in isolation. Unit ini mungkin mengandung beberapa logika bisnis yang perlu diuji secara terpisah. When writing isolated unit tests all external dependencies should be mocked out, eg using Spies . Saat menulis unit test yang terisolasi, semua dependensi eksternal harus diejek, misalnya menggunakan Mata - mata .

Here is an example of a service with an isolated test: Berikut adalah contoh layanan dengan tes terisolasi:

import { TestBed, inject } from '@angular/core/testing';
import { TodoListService } from './todo-list.service';
import { HttpClient } from '@angular/common/http';
import { TODOItem } from '../shared/models/todo-item';
import { of } from 'rxjs/observable/of';
describe('Service: TodoList', () => {
let todoListService: TodoListService;
const httpClientSpy: jasmine.SpyObj<HttpClient> = jasmine.createSpyObj('httpClient', ['get']);
httpClientSpy.get.and.returnValue(of([new TODOItem('Buy Milk', 'Lala')]));
beforeEach(() => {
TestBed.configureTestingModule({
providers: [TodoListService,
{
provide: HttpClient,
useValue: httpClientSpy
}]
});
});
beforeEach(inject([TodoListService], (service: TodoListService) => {
todoListService = service;
}));
it('should be defined', () => {
expect(todoListService).toBeTruthy();
});
it('should get todoList using http request', () => {
expect(httpClientSpy.get).toHaveBeenCalled();
expect(todoListService.todoList.length).toBe(1);
});
});

Shallow unit testing Pengujian unit dangkal

Shallow testing is when you test a component with a template but you don't render child component by setting schema to NO_ERRORS_SCHEMA. Pengujian dangkal adalah saat Anda menguji komponen dengan templat tetapi Anda tidak merender komponen turunan dengan menetapkan skema ke NO_ERRORS_SCHEMA. This will ignore unknown tags in the template, making us not needing to import the child components. Ini akan mengabaikan tag yang tidak dikenal di templat, membuat kami tidak perlu mengimpor komponen turunan. An example of a shallow tested component: Contoh komponen yang diuji dangkal:

Integration testing Tes integrasi

Integration testing is when you test two or more components together. Pengujian integrasi adalah saat Anda menguji dua komponen atau lebih secara bersamaan. This makes sense for parts of the applications where the integration between component is important to test, such as a smart/container component. Ini masuk akal untuk bagian aplikasi di mana integrasi antara komponen penting untuk diuji, seperti komponen smart / container. With integration testing, you simply import the implementation of the dependencies of the unit under test even though you still should mock out HTTPClient. Dengan pengujian integrasi, Anda cukup mengimpor implementasi dependensi unit yang sedang diuji meskipun Anda masih harus mengejek HTTPClient. End to end testing Pengujian ujung ke ujung

End to end testing means that you run the complete application together, including the backend, using a browser automatization API, such as Selenium. Pengujian ujung ke ujung berarti Anda menjalankan aplikasi lengkap bersama-sama, termasuk backend, menggunakan API otomatisasi browser, seperti Selenium. Protractor, which is built on top of Selenium, ships with an Angular CLI project and is what we will be working with here. Protractor, yang dibangun di atas Selenium, dikirimkan dengan proyek CLI Angular dan akan kami kerjakan bersama di sini.

Page object Objek halaman

Out of the box a new Angular CLI project ships with the page object pattern , which is a method of separating the stable and the fragile part of the end to end testing into a .po file (fragile part) and the .e2e-spec file (stable part). Di luar kotak, proyek CLI Angular baru dikirimkan dengan pola objek halaman , yang merupakan metode untuk memisahkan bagian stabil dan rapuh dari pengujian ujung ke ujung menjadi file .po (bagian rapuh) dan file .e2e-spec (bagian stabil).

import { browser, by, element } from 'protractor';
export class AppPage {
navigateTo() {
return browser.get('/');
}
getAmountOfTODOs() {
return element(by.css('.todo-item')).count;
}
}

End to end test spec Spesifikasi tes ujung ke ujung

The test specification should use the page object for accessing dom elements. Spesifikasi pengujian harus menggunakan objek halaman untuk mengakses elemen dom. Protractor comes with built-in integrations for Angular such as waitForAngular, which will wait for an Angular page to be ready before doing something. Protractor hadir dengan integrasi bawaan untuk Angular seperti waitForAngular, yang akan menunggu halaman Angular siap sebelum melakukan sesuatu.

import { browser } from 'protractor';
import { AppPage } from './app.po';
describe('todoapp App', () => {
let page: AppPage;
beforeEach(() => {
page = new AppPage();
});
it('should display welcome message', (done) => {
page.navigateTo();
browser.waitForAngular().then(() => {
expect(page.getAmountOfTODOs()).toBe(2);
done();
});
});
});

A word about fragile E2E tests Sepatah kata tentang tes E2E yang rapuh

If you have had any experience with automatic E2E testing you know that they are fragile by nature and breaks easily. Jika Anda pernah memiliki pengalaman dengan pengujian E2E otomatis, Anda tahu bahwa mereka rapuh secara alami dan mudah rusak.

Here are some techniques to make them more stable: Berikut adalah beberapa teknik untuk membuatnya lebih stabil:

  • Retry expects with a reasonable fallback and retry count, eg. Coba lagi harapkan dengan fallback dan coba lagi masuk akal, misalnya. 1 second retry of up to 5 retries Coba lagi 1 detik hingga 5 percobaan lagi
  • Automatically rerun tests that have a fragile dependency, such as a fragile dev API Secara otomatis jalankan kembali tes yang memiliki ketergantungan rapuh, seperti API dev rapuh
  • Make page objects select using id attributes since classes are more prone to changes Buat objek halaman pilih menggunakan atribut id karena kelas lebih rentan terhadap perubahan

A pragmatic standpoint on writing Angular tests Sudut pragmatis dalam menulis tes Angular

Writing tests for Angular/front end apps is not the same as writing tests for back end systems. Tes penulisan untuk aplikasi sudut / ujung depan tidak sama dengan tes penulisan untuk sistem ujung belakang. Why? Mengapa? With back end testing you don't have a fragile and slow GUI to interact with but just plain server code. Dengan pengujian ujung belakang Anda tidak memiliki GUI yang rapuh dan lambat untuk berinteraksi dengan tetapi hanya kode server biasa. When you are testing an Angular app it is hard to automatically prove that it looks and works correctly, as the requirements are softer than clearly defined business logic. Ketika Anda menguji aplikasi Angular, sulit untuk secara otomatis membuktikan bahwa itu terlihat dan berfungsi dengan benar, karena persyaratannya lebih lunak daripada logika bisnis yang didefinisikan dengan jelas.

For this reason, I recommend NOT to have a 100% percent code coverage as not all tests give an equal return of investment and we want to focus our effort on the code that is closest to the use cases and business logic which is going to yield the highest ROI. Untuk alasan ini, saya sarankan TIDAK untuk memiliki cakupan kode 100% karena tidak semua tes memberikan pengembalian investasi yang sama dan kami ingin memfokuskan upaya kami pada kode yang paling dekat dengan kasus penggunaan dan logika bisnis yang akan menghasilkan ROI tertinggi.

I recommend prioritizing your tests effort like this: Saya sarankan memprioritaskan upaya pengujian Anda seperti ini:

  • Services – Sandboxes , business logic, NgRx effects (100% coverage) Layanan - Kotak Pasir , logika bisnis, efek NgRx (cakupan 100%)
  • Pure functions – Pipes, NgRx reducers , helpers (100% coverage) Fungsi murni - Pipa, reduksi NgRx , pembantu (cakupan 100%)
  • Container component – Integration tests for happy paths (mock out services but render all components) Komponen kontainer - Tes integrasi untuk jalur senang (layanan tiruan tapi render semua komponen)

Having these tests in place will yield you the highest ROI. Memiliki tes ini di tempat akan menghasilkan Anda ROI tertinggi. When starting to write unit tests for presentation components, the ROI is getting less and less because the tests are likely not going to save you if your component looks wrong. Saat mulai menulis unit test untuk komponen presentasi, ROI semakin berkurang karena tes kemungkinan tidak akan menyelamatkan Anda jika komponen Anda terlihat salah.

What about presentation component? Bagaimana dengan komponen presentasi?

Presentation components are, as the name implies, about the UI of the component. Komponen presentasi, sesuai namanya, tentang UI komponen. It is hard to automatically verify changes in the UI. Sulit untuk memverifikasi perubahan secara otomatis di UI. I have tried a couple of solutions already such as plugins for Selenium and SaaS solutions and all of them had been full of flakiness and false positives. Saya sudah mencoba beberapa solusi seperti plugin untuk Selenium dan solusi SaaS dan semuanya sudah penuh dengan keripik dan positif palsu.

Using Jest's snapshot tests might be the best option here to test presentation components, but still, manual testing will be needed to make sure everything looks and works correctly. Menggunakan tes snapshot Jest mungkin merupakan opsi terbaik di sini untuk menguji komponen presentasi, tetapi tetap saja, pengujian manual akan diperlukan untuk memastikan semuanya terlihat dan berfungsi dengan benar.

Keeping unit tests fast by running them separately Menjaga unit test cepat dengan menjalankannya secara terpisah

Running unit tests should be fast, so you get the fastest feedback cycle possible when doing TDD style development. Tes unit yang berjalan harus cepat, sehingga Anda mendapatkan siklus umpan balik tercepat saat melakukan pengembangan gaya TDD. The bottleneck for Angular test executing is usually component template rendering. Hambatan untuk mengeksekusi tes Angular biasanya rendering templat komponen. For this reason, I recommend postfixing integration test files with integration.spec.ts for distinguishing between unit tests (.spec.ts postfix). Untuk alasan ini, saya merekomendasikan file tes integrasi postfixing dengan integration.spec.ts untuk membedakan antara unit test (.spec.ts postfix). I use this for creating NPM scripts for running different kinds of test: all tests, unit tests only and integration tests only. Saya menggunakan ini untuk membuat skrip NPM untuk menjalankan berbagai jenis tes: semua tes, hanya tes unit dan hanya tes integrasi.

Setup config for unit tests Konfigurasi konfigurasi untuk pengujian unit

For running unit test you want to run all test that ends with .spec and exclude all .integration.spec.ts: Untuk menjalankan tes unit, Anda ingin menjalankan semua tes yang berakhir dengan .spec dan mengecualikan semua .integration.spec.ts:

// This file is required by karma.conf.js and loads recursively all the .spec and framework files
import { getTestBed } from '@angular/core/testing';
import { BrowserDynamicTestingModule, platformBrowserDynamicTesting } from '@angular/platform-browser-dynamic/testing';
import 'hammerjs';
import 'jquery';
import 'materialize-css';
// Unfortunately there's no typing for the `__karma__` variable. Just declare it as any.
declare const __karma__: any;
declare const require: any;
// Prevent Karma from running prematurely.
__karma__.loaded = function() {};
// First, initialize the Angular testing environment.
getTestBed().initTestEnvironment(BrowserDynamicTestingModule, platformBrowserDynamicTesting());
// Then we find all the tests.
const context = require.context('./', true, /\.spec\.ts/);
// And load the modules.
context
.keys()
.filter(function(element, index) {
// The regex in require.context didn't work for filtering integration testing off
return !element.endsWith('.integration.spec.ts');
})
.map(context);
// Finally, start Karma to run the tests.
__karma__.start();

Discovering slow tests Menemukan tes lambat

You can use Karma Verbose Reporter to measure the execution speed for each test and postfix them with integration.spec.ts if they are too slow so they won't impact unit test, that always should be running fast. Anda dapat menggunakan Karma Verbose Reporter untuk mengukur kecepatan eksekusi untuk setiap tes dan mempostingnya dengan integrasi.spec.ts jika mereka terlalu lambat sehingga mereka tidak akan memengaruhi tes unit, yang selalu harus berjalan cepat.

Wrapping up Membungkus

In this post, we discussed the different ways of testing an Angular app: isolated, shallow, integration, end to end and visual regression testing . Dalam posting ini, kami membahas berbagai cara menguji aplikasi Angular: terisolasi, dangkal, integrasi, ujung ke ujung dan pengujian regresi visual .
We looked at a pragmatic approach to testing to focus on services, pure functions, and container components to ensure the highest ROI when testing. Kami melihat pendekatan pragmatis untuk pengujian untuk fokus pada layanan, fungsi murni, dan komponen kontainer untuk memastikan ROI tertinggi saat pengujian.
We also saw how to keep unit tests fast by using a separate postfix for integration test spec files, setting up NPM scripts for running different types of tests and how to discover slow tests that should be moved to integration tests. Kami juga melihat bagaimana menjaga tes unit tetap cepat dengan menggunakan postfix terpisah untuk file spesifikasi tes integrasi, mengatur skrip NPM untuk menjalankan berbagai jenis tes dan bagaimana menemukan tes lambat yang harus dipindahkan ke tes integrasi.
